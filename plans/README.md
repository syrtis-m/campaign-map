# Implementation Plans — toward a first-class map builder

Generated by the `improve` skill on 2026-07-09, planned against commit `3783bf9`.

These plans target the three pain points raised for moving the plugin from
"half-baked" to a first-class map builder: (1) no full UX, (2) map dots too
small to click, (3) the background/basemap is hard to swap / generate / create
from map data. Read each plan fully before starting; honor its STOP conditions;
update your status row when done.

| 003 | On-map control surface (visible toolbar replacing command-palette-only UX) | P2 | M | — | TODO |

All three are independent and can ship in any order. Recommended order is by
leverage: **001 first** (smallest, highest daily-annoyance payoff, zero risk to
other systems), then **002** (biggest visual "it's a real map now" jump, pure
render-layer change), then **003** (the broadest UX lift).

## Why these three, and not a single "build the UX" plan

The "lack of a full UX" pain point is real but too large for one executable
plan. It decomposes into concrete, independently-shippable pieces; 001 and 003
are two of them (clickable features; a visible control surface). The remaining
UX arc is captured as **Direction** below rather than as a monster plan a weak
executor can't act on.

## Dependency notes

- No hard dependencies between the three plans. If both 001 and 003 are done,
  the "generated settlement → make canon" affordance should live in exactly one
  of them — 001 owns the *click routing* for generated points; 003 owns any
  *toolbar button*. They don't conflict.
- 002 is unblocked because the generator **already** attaches `biome`, `height`,
  and `moisture` to `world-region` feature properties
  (`src/gen/world/regions.ts:25`) — so it is a pure render-layer change with **no
  generator edits and therefore no new 2×2 seam tests required** (CLAUDE.md's
  seam-test mandate applies to `src/gen/` changes; this plan touches none).

## Direction — the rest of the "first-class map builder" arc

These are options for the maintainer to weigh, not ranked problems. Each is
grounded in current code.

1. **Empty-state / onboarding on the map canvas.** A brand-new campaign opens to
   a blank themed box with a scale bar; every next action (generate world,
   generate city, add a location) is hidden in the command palette
   (`src/main.ts:165-219`). A first-class builder greets an empty campaign with
   a visible "Generate a world here" / "Add your first location" call to action
   on the canvas. Cheap once 003's DOM-overlay pattern exists; high first-run
   impact. *(Design/spike, then a small build.)*

2. **In-app basemap acquisition for real cities.** Today swapping a real-city
   background requires running the `pmtiles` Go binary in a terminal by hand
   (`src/view/CampaignControlModal.ts:144-206` only *generates the command*). The
   architecture deliberately forbids the plugin executing binaries, but the
   friction is real. Option: ship a documented, one-screen guided flow (detect
   whether `pmtiles` is on PATH, pre-fill bbox from the viewport, show progress
   instructions) — or investigate a pure-JS tile source as a fallback basemap so
   "swap the background" never requires leaving Obsidian. *(Spike: what's the
   lightest real-city background that needs zero external binary?)*

3. **Generated-fabric as a first-class, browsable layer with a legend.** World
   generation computes rich data (heightmap, moisture, biomes, regions,
   settlements, routes) but the map exposes almost none of it to the GM as
   *controllable* layers — no legend, no per-layer visibility toggles, no way to
   see "what biome is this / what region am I in." Plan 002 paints the terrain;
   the direction extension is a layer panel + legend that makes the generated
   world explorable, not just decorative. *(Builds naturally on 002 + 003.)*

4. **Turn generated features into editable canon in bulk.** Canonizing is
   one-feature-at-a-time via `canonize-nearest-generated`
   (`src/view/MapView.ts:591`). A world generates dozens of settlements; a
   builder wants "canonize this whole region's settlements" or "canonize the
   visible city." The data path already exists (`canonizeFeature`), so this is
   mostly UX + a batch loop. *(Small build once 003 gives it a home.)*

## Known findings noted but NOT planned here

- **Post-session renderer perf / state leak** (PROGRESS.md, Phase 4 notes +
  2026-07-08 data point): reported p95 fps degrades over a long-lived Obsidian
  session and only a full window reload restores it; `plugin:reload` alone does
  not. Real, logged, not root-caused. Left unplanned because it needs profiling
  (an *investigation* plan, not a fix plan) — surface it here so it isn't lost.
  Recommend a `deep perf` audit pass focused on retained timers/`Map` entries in
  the viewport dispatcher and workspace-layer state if it starts biting.

## Findings considered and rejected

- *Enlarge the visible dots to make them easier to click*: rejected as the fix
  for 001. It fights the product bar (label collisions, "cartographic discipline
  — never push styling onto the GM"). The correct fix is an invisible hit
  tolerance, not fatter dots. (See 001.)
